## 워크-포워드 분석이 중요한 이유

크립토 시장은 변합니다. 한 레짐에서 잘 작동하던 방법이 다음 레짐에서는 실패하는 일이 흔합니다. 단일한 "최적" 파라미터 집합은 취약합니다 — 곡선 맞춤(curve-fitting), 데이터 스누핑, 우연의 결과일 가능성이 높습니다.

워크-포워드 분석(WFA)은 새로운 데이터가 들어올 때마다 재학습(re-training)하고 재검증하는 현실적인 과정을 역사적 기록에 모방해 측정하는 검증 방법입니다. 단일 훈련/테스트 분할 대신, WFA는 롤링 윈도우에서 반복적으로 재최적화를 수행하고 이후 기간에 대한 성과를 기록합니다. 결과적으로 얻는 아웃오브샘플 성과 분포는 전략이 견고한지, 단지 과거 데이터에 맞춰진 것인지 분간하게 해줍니다.

실거래 성과와 여러 레짐에서의 생존 가능성에 신경 쓴다면, 워크-포워드 분석은 필수 검증 도구입니다.

## 핵심 용어(요약)

- 인-샘플(IS): 파라미터 최적화에 사용되는 훈련 구간입니다.
- 아웃오브샘플(OOS) / 포워드 테스트: IS 바로 뒤에 오는 홀드아웃 구간으로 성능을 평가합니다.
- 워크-포워드 스텝: IS에서 최적화하고 OOS에서 테스트하는 한 사이클입니다.
- 롤링 윈도우: IS와 OOS 윈도우를 시간축으로 앞으로 밀며 반복하는 방식입니다.

## 핵심 아이디어(직관적으로)

1. 훈련 윈도우(예: 12개월)와 포워드 테스트 윈도우(예: 1개월)를 정합니다.
2. 훈련 기간에서 파라미터를 최적화합니다(오직 IS 데이터만 사용).
3. 최적화된 파라미터로 포워드 기간에서 테스트하고 결과를 기록합니다.
4. IS와 OOS를 포워드 기간만큼 이동하고 반복합니다.
5. 모든 포워드 테스트 결과를 집계해 안정성과 일반화 가능성을 평가합니다.

이는 과거 데이터에 전략을 맞추고 실제로 배포했을 때 어떤 결과가 나왔을지를 역사적으로 시뮬레이션하는 방식입니다.

## 단계별 워크-포워드 절차

다음은 프로그램적 백테스트나 PRUVIQ의 Strategy Builder + 프리셋을 사용해 실행할 수 있는 실용적인 워크플로입니다.

1) 윈도우와 스트라이드 정의

- IS_length와 OOS_length를 정합니다. 예: IS = 12개월, OOS = 1개월.
- 스트라이드는 보통 OOS_length(겹치지 않음) 또는 더 짧게 설정해 표본 수를 늘립니다(예: 주단위).

팁: 재학습 주기를 실제 운영 주기와 맞추세요(월간 재최적화 → OOS = 1개월).

2) 작고 합리적인 파라미터 그리드 선택

자유 파라미터가 많으면 과적합 위험이 커집니다. 그리드는 거칠게 설정하세요. 예: 

- RSI_threshold: [25, 30, 35]
- EMA_length: [20, 50]
- SL: [1%, 1.5%, 2%]

3) 각 워크-포워드 스텝에서

- IS 데이터만 사용해 파라미터 그리드를 최적화합니다. 목적 함수는 기대값(expectancy) 또는 프로핏 팩터에 드로우다운 제약을 추가하는 등 단순하면서도 보수적이어야 합니다.
- 최적 파라미터를 선택하고 기록합니다.
- 선택된 파라미터로 OOS에서 백테스트를 실행하고 메트릭(순손익, 거래 수, 프로핏 팩터, 최대 드로우다운, 기대값)을 저장합니다.

4) 역사 범위가 소진될 때까지 반복

윈도우를 앞으로 밀면서 모든 OOS 결과를 수집합니다.

5) 집계 및 평가

모든 OOS 스텝에서 다음 요약 지표를 계산합니다:

- 합격률(Pass rate): OOS 스텝 중 순기대값이 양수인 비율
- 중앙값/평균 OOS 수익 및 프로핏 팩터
- 관찰된 최대 OOS 드로우다운
- 파라미터 생존성(각 파라미터 값이 선택된 빈도)

6) 수용 기준 결정

정답은 없지만 실무에서 자주 사용하는 시작점 예시는 다음과 같습니다:

- 합격률 >= 60–75% (기간과 표본에 따라 조정)
- 중앙값 OOS 기대값 > 0
- 프로핏 팩터가 스텝 간 크게 흔들리지 않음
- 파라미터가 완전히 무작위로 바뀌지 않고 어느 정도 집중됨(생존성)

이 기준을 만족하지 못하면 전략이 과적합이거나 레짐 의존적일 가능성이 큽니다.

## 의사코드(Pseudocode)

```
for start in range(t0, t_end - IS_length - OOS_length, stride):
    IS = data[start : start + IS_length]
    OOS = data[start + IS_length : start + IS_length + OOS_length]

    best_params = optimize(IS, param_grid)
    metrics = backtest(OOS, best_params)
    record(metrics, best_params)

aggregate_results = summarize(records)
```

단순한 형태입니다 — 네스티드 CV, 확장 윈도우(expanding window), 다른 목적함수 등 다양한 변형이 가능합니다. 핵심은 위 루프입니다.

## PRUVIQ 도구를 활용한 실전 예시

PRUVIQ의 Strategy Builder는 프리셋 저장과 파라미터 스윕을 지원합니다. 이를 WFA의 구성요소로 사용할 수 있습니다(빌더 안내 참고: /blog/strategy-builder-step-by-step 또는 /simulate 열기). 기본 접근 방식은 다음과 같습니다:

1. 전략의 기본 프리셋을 만듭니다(진입/청산 규칙, 사이징).
2. 빌더의 스윕 기능(또는 반복 실행)을 사용해 IS 윈도우에서 파라미터를 최적화합니다.
3. IS에서 선택한 최적 프리셋을 저장하고 OOS 윈도우에서 다시 실행합니다(프리셋 로드 후 히스토리 범위 지정).
4. 자동화 또는 스크립트로 루프를 실행하려면 API를 호출해 결과를 수집하세요. (OHLCV 엔드포인트와 프리셋 기능은 플랫폼에서 백엔드로 제공됩니다 — 빌더 가이드 참조: src/content/blog/strategy-builder-step-by-step.md)

참고: 빌더는 대화형 테스트와 빠른 확인에 적합합니다. 워크-포워드 스텝을 많게 돌려야 한다면 API로 자동화하는 것이 효율적입니다.

## 추적할 메트릭(각 OOS 스텝)

- 순손익(Net P&L)
- 거래 수(샘플 크기)
- 거래당 기대값(Expectancy)
- 프로핏 팩터
- 최대 드로우다운
- IS에서 선택된 파라미터 집합

이들을 통해 평균/중앙값/표준편차를 계산하고 박스플롯, 파라미터 생존성 히트맵 같은 시각화를 만드세요.

## WFA와 함께 사용할 고급 검증

- 몬테카를로 트레이드 셔플: 트레이드 순서를 무작위화하거나 리턴을 재샘플링해 관찰된 엣지가 우연인지 아닌지 평가
- WFA + 몬테카를로: 각 OOS 스텝에서 몬테카를로를 적용해 신뢰구간을 얻음
- 비용 민감도 테스트: 슬리피지/수수료를 증가시켜 전략이 작은 비용 변화에 취약한지 확인
- 코인 간 일반화 테스트: 여러 코인에서 WFA를 실행해 시그널이 특정 코인에만 국한되지 않는지 확인(빌더의 크로스-코인 백테스트 사용 권장)

## 파라미터 생존성 해석

파라미터 생존성은 실무에서 유용한 안정성 지표입니다. 같은 파라미터 값(또는 좁은 범위)이 IS 윈도우에서 반복적으로 선택된다면 신호의 안정성이 높다는 신호입니다. 반대로 매 스텝마다 선택값이 널뛰면 노이즈일 가능성이 큽니다.

권장 방법:

- 각 이산 파라미터 값의 선택 빈도를 센다.
- 막대그래프로 시각화한다. 어떤 값이 전체 스텝의 >40–50%에 나타나면 "생존자"로 간주한다. <10%는 노이즈 가능성.

(임계값은 경험적 규칙입니다 — 전략과 표본 크기에 맞춰 조정하세요.)

## 흔한 실수와 회피 방법

- 자유 파라미터가 너무 많음: 그리드를 줄이고 거친 범위를 선호하세요.
- 너무 작은 OOS 창: OOS에서 의미 있는 평가를 하려면 충분한 거래가 필요합니다(평균 거래 수가 <10이면 노이즈가 큼).
- 실행 현실성 무시: 항상 보수적인 슬리피지/수수료를 시뮬레이션하세요(빌더 기본값 사용 권장).
- 윈도우를 조작하거나 결과를 훑어본 뒤 파라미터를 변경하는 행위: 윈도우와 스트라이드는 테스트 전에 고정하세요.

## 운영적 의사결정 흐름(예시)

1. WFA를 실행해 역사 범위를 모두 통과시킨다.
2. 합격률이 임계값 미만이거나 중앙 OOS 기대값 <= 0이면 전략을 거부하거나 재설계한다.
3. 합격률은 괜찮지만 드로우다운이 가끔 크게 발생하면 리스크 컨트롤을 강화한다(포지션 축소, 손절 조정, 레버리지 제한).
4. WFA를 충분히 통과하면 페이퍼/소액 실거래로 포워드 테스트를 실행한다. 모든 시그널을 로깅하고 실제 체결을 시뮬레이션과 비교한다.

포워드 테스트가 백테스트와 일치할 때 비로소 배포를 고려하세요.

## 결론

워크-포워드 분석은 하나의 낙관적인 백테스트를 시간에 따른 일련의 현실적 포워드 결과로 바꿉니다. 안정성, 파라미터 생존성, 레짐 민감도를 노출시키며 이는 크립토 시장처럼 비정상성(non-stationary)이 강한 환경에서 특히 중요합니다.

대화형 IS/OOS 테스트는 Strategy Builder로 빠르게 하고, 많은 스텝이 필요하면 API로 자동화하세요. WFA를 몬테카를로와 비용 민감도 테스트와 결합하면 실제 배포 전에 훨씬 높은 신뢰를 얻을 수 있습니다.

기본 백테스트 규칙과 선행 편향(look-ahead) 함정은 다음 글들을 참고하세요: /ko/blog/how-to-backtest-crypto-strategy 및 /ko/blog/why-backtests-lie.
