# AUTONOMY.md — Automation & Governance for PRUVIQ

This document describes the automation policy, merge rules, versioning, and rollback procedures for the PRUVIQ project. Keep this file small and human-readable — it is the authoritative runbook for automated agents.

## 1. High-level intent
- Agents (프루빅 및 내부 에이전트들) are authorized to create branches, open PRs, run CI, and propose merges.
- Changes should be small, incremental, and safe. Agents must prefer conservative, reversible edits and provide clear evidence for their changes.

## 2. Versioning
- Use a minimal patch-only versioning scheme for agent-driven changes: start at `0.0.1` and increment the PATCH (third) component for each release or set of automated changes (0.0.1 → 0.0.2 → ...).
- Tag the repo with `v<version>` when merging to main (e.g., `git tag v0.0.1`).

## 3. Merge policy (default)
- Default automation mode: LABEL‑BASED AUTOMERGE (recommended & enabled)
  - PRs labelled `automerge` will be eligible for automatic merging if and only if:
    - All required CI checks pass (build, tests, a11y, security scans, startup-file validation).
    - The PR does not modify any backend or restricted files (see section 6).
    - The PR description includes the mandatory PR template (What / Why / Result / Next) and a version bump if releasing.
  - If the PR meets conditions, the automation may merge and wait for post-deploy smoke checks.

- Manual override: If a reviewer (JEPO or repo admin) sets `force-merge`, the PR may be merged manually following review.

## 4. Post-merge & deploy
- After merge, Cloudflare Pages auto-deploys. The automation must run smoke checks (curl core pages + health endpoints). If smoke checks fail, create a revert PR and notify ops immediately.

## 5. Rollback policy
- If a post-deploy smoke check fails, the agent must:
  1. Open a revert PR that reverts the merge commit.
  2. Post an incident comment with failure evidence (logs, curl output, timestamps).
  3. Notify JEPO/ops in the configured channel.

## 6. Restricted areas
- Agents MUST NEVER modify backend/ files or any directories marked `READ ONLY` in SOUL.md.
- Sensitive secrets, keys, and credentials must never be committed — use GitHub Actions secrets or Cloudflare environment variables. Agents will create documentation/runbooks requesting ops to provision them.

## 7. CI guards
- Required checks for automerge (examples):
  - `npm run build` (no errors)
  - Playwright E2E (non-blocking for minor docs, blocking for UI changes)
  - Axe accessibility scan (a11y)
  - `validate-startup-files` (ensures WORKFLOW_AUTO.md present and summarizer runs)
  - security scan (npm audit or Snyk)

## 8. Reporting
- Every agent action that changes repository state must append a short line to `MEMORY.md` and include a PR body with `What / Why / Result / Next`.
- Weekly agent performance summary (automatically generated) will list success_rate, human_intervention_rate, and PR_accept_rate.

## 9. Safe defaults
- Use conservative defaults: automerge only for `automerge` labeled PRs; otherwise await human approval.
- For critical changes (infra, domain, secrets), require explicit JEPO approval via PR review.
