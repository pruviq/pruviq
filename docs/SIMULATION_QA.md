## 0. 이 문서의 목적

PRUVIQ는 사용자에게 전략 시뮬레이션 결과를 제공하는 플랫폼이다. 그 결과가 **잘못되면** 사용자가 실제 돈을 잃는다. 따라서 시뮬레이션 결과의 정확성과 신뢰성은 PRUVIQ의 생사를 결정하는 핵심 요소다.

이 문서는 창업자(이재풍)가 autotrader 2개월 운영 과정에서 직접 겪은 실패 사례를 기반으로, 같은 실수를 반복하지 않기 위한 체계적 검증 프레임워크를 설계한다.

### autotrader에서 배운 핵심 교훈 (이 문서의 존재 이유)

## 1. 시뮬레이션 정확성 검증 (Accuracy Verification)

### 1.1 Look-ahead Bias 자동 탐지

Look-ahead bias는 시뮬레이션에서 가장 치명적인 오류이다. autotrader에서 2026-01-10에 발견한 이후 모든 시뮬레이션 코드에 구조적 방어가 필요하다.

#### 1.1.1 구조적 방어: 시그널-진입 분리 원칙

```
절대 규칙:
  1. check_signal(df, idx) → idx 행까지의 데이터만 사용 가능
  2. 진입가 = df.iloc[idx + 1]["open"] (다음 캔들 시가)
  3. SL/TP 체크 = 진입 이후 캔들만 확인
  4. 지표 계산에 shift(n) 사용 시 n > 0 (과거 방향만 허용)
```

#### 1.1.2 자동 탐지 테스트 체계 (5종)

**Test A: 시간 순서 검증 (Time Order Test)**
- 모든 거래의 exit_time >= entry_time 확인
- 모든 거래의 entry_time > signal_time 확인 (시그널 봉 이후)
- bars_held >= 0 확인

**Test B: 진입가 일관성 검증 (Entry Price Test)**
- entry_price == df.iloc[entry_idx]["open"] (슬리피지 조정 전) 확인
- entry_idx == signal_idx + 1 확인

**Test C: 미래 데이터 접근 차단 테스트 (Future Data Access Test)**
- 전략의 check_signal(df, idx) 호출 시 df를 df.iloc[:idx+1]로 잘라서 전달
- 잘린 데이터에서도 동일한 시그널이 발생하는지 확인
- 불일치 발생 → 미래 데이터 접근 탐지

**Test D: 결과 방향성 검증 (Directional Sanity Test)**
- 랜덤 전략 (매 봉 50% 확률 진입) 시 수수료 차감 후 기대 수익률 < 0 확인
- 만약 랜덤 전략의 수익률 > 0 → 엔진 자체에 bias 존재

**Test E: 시프트 패턴 분석 (Shift Pattern Audit)**
- 전략 코드에서 `shift(-n)` (n > 0) 패턴 자동 탐지 → 미래 방향 shift 금지
- `df.iloc[idx + k]` (k > 0) 패턴이 check_signal 내부에 존재하면 경고
- 지표 계산에서 `.rolling().mean()` 등의 center=True 사용 탐지

#### 1.1.3 Strategy Protocol에 강제하는 계약

```
모든 전략 구현체는 다음을 보장해야 한다:

1. check_signal(df, idx)에서 df.iloc[idx+1:]에 접근하지 않음
2. calculate_indicators(df)에서 미래 정보를 사용하는 지표를 계산하지 않음
   - rolling의 center=True 금지
   - shift(-n) 금지 (n > 0)
   - fillna(method='bfill') 금지 (미래 값으로 채움)
3. get_params()가 모든 파라미터를 빠짐없이 반환함 (캐시 키 정확성)
```

### 1.2 캔들 인덱스 버그 방지 (Entry Timing Verification)

autotrader에서 2026-01-31에 발생한 사건: `prev` vs `prev2` 캔들 비교 1개 차이로 -20.6% vs +$794의 극단적 차이가 발생했다.

#### 1.2.1 인덱스 규약 정의

```
PRUVIQ 캔들 인덱스 규약:

시점: 10:01 UTC (캔들 교체 직후)
  - df.iloc[idx]     = "prev" = 09:00-10:00 캔들 (완성됨, 사용 가능)
  - df.iloc[idx + 1] = "curr" = 10:00-??:?? 캔들 (진행 중, open만 사용 가능)
  - df.iloc[idx - 1] = "prev2" = 08:00-09:00 캔들 (완성됨, 사용 가능)

엔진 규칙:
  - 시그널 판단: idx 행까지의 데이터 (prev)
  - 진입: idx + 1의 open (curr의 시가)
  - SL/TP 체크: idx + 1 이후 봉의 high/low
  - TIMEOUT 청산: idx + max_bars의 close
```

#### 1.2.2 인덱스 일관성 테스트

- 동일 전략을 1봉 shift한 데이터에서 실행 → 거래 수와 타이밍이 정확히 1봉 shift되는지 확인
- SL/TP 가격이 entry_price 기준으로 정확히 계산되는지 검증 (부동소수점 오차 허용: 1e-10)
- 동일 봉에서 SL + TP 동시 도달 시 보수적 처리(SL 우선) 확인

### 1.3 비용 모델 정확성 검증

#### 1.3.1 비용 모델 검증 항목

| 항목 | 검증 방법 | 허용 오차 |
|------|----------|----------|
| 거래소 수수료율 | 거래소 공식 문서와 대조 | 0% (정확 일치) |
| 슬리피지 모델 | 실거래 데이터 대비 비교 (autotrader 데이터 참고) | +-50% |
| 펀딩비 (선물) | 거래소 히스토리 API 데이터 대조 | +-10% |
| 총 비용 합리성 | 거래당 평균 비용이 합리적 범위 내 | 현물 0.1-0.3%, 선물 0.05-0.15% |

#### 1.3.2 비용 모델 투명성 요구사항

모든 시뮬레이션 결과에 다음을 명시해야 한다:

```
비용 모델 명세:
  - 거래소: Binance Futures
  - Maker 수수료: 0.02%
  - Taker 수수료: 0.04%
  - 적용 수수료: 0.04% (Taker 가정, 보수적)
  - 슬리피지: 0.02% (고정)
  - 펀딩비: 미반영 / 반영 (8시간 기준)
  - 1회 왕복 총 비용: ~0.12%
```

#### 1.3.3 비용 감도 분석 (Cost Sensitivity Test)

모든 전략에 대해 비용 0%, 1x, 2x, 3x를 적용한 결과를 비교:
- 비용 0%에서만 수익인 전략 → "비용에 취약한 전략" 경고
- 비용 3x에서도 수익인 전략 → "비용 내성 높은 전략" 표시

### 1.4 결과 재현성 보장 (Reproducibility)

#### 1.4.1 결정론적 실행 보장

```
재현성 요건:
  - 동일한 (전략 + 파라미터 + 데이터 + 비용모델) → 동일한 결과
  - 랜덤 요소가 있는 전략 → random seed 필수 기록
  - 병렬 실행 순서와 무관한 결과 (코인 간 독립)
```

#### 1.4.2 재현성 테스트

- 동일 입력으로 시뮬레이션 2회 실행 → 비트 단위 동일 결과 확인
- 다른 머신(MacBook vs Mac Mini)에서 실행 → 부동소수점 차이 허용 범위 내 (1e-6) 확인
- 캐시 히트와 실제 실행 결과 비교 → 일치 확인

#### 1.4.3 결과 해시 서명

```
모든 시뮬레이션 결과에 포함:
  - result_hash: SHA-256(전략명 + 파라미터 + 데이터 버전 + 결과 요약)
  - data_version: 데이터 최종 타임스탬프
  - engine_version: SimulationEngine 버전
  - strategy_version: 전략 코드 버전
  - timestamp: 시뮬레이션 실행 시각
```

## 2. 통계적 신뢰성 (Statistical Reliability)

### 2.1 최소 거래 수 기준

"30건으로 유의미한 결론을 내릴 수 있다"는 통계학의 CLT(중심극한정리)에 기반하지만, 금융 시계열은 정규분포가 아니므로 더 보수적인 기준이 필요하다.

#### 2.1.1 PRUVIQ 최소 거래 수 등급

| 등급 | 거래 수 | 표시 | 신뢰도 | 사용자에게 표시 |
|------|---------|------|--------|----------------|
| **Insufficient** | < 30 | 빨간색 경고 | 매우 낮음 | "거래 수 부족. 결과를 신뢰하기 어렵습니다." |
| **Preliminary** | 30-99 | 주황색 주의 | 낮음 | "초기 결과. 추가 데이터로 검증이 필요합니다." |
| **Moderate** | 100-299 | 노란색 | 보통 | "보통 신뢰도. 통계적 유의성 확인 중." |
| **Reliable** | 300-999 | 연두색 | 높음 | "높은 신뢰도. 통계적으로 유의합니다." |
| **Highly Reliable** | 1000+ | 초록색 | 매우 높음 | "매우 높은 신뢰도. 충분한 표본." |

#### 2.1.2 신뢰 등급 산정 기준

단순 거래 수만이 아니라 다음을 종합 고려:

```
신뢰 점수 = f(거래수, 기간길이, 시장상태 다양성, 코인 다양성)

가중치:
  - 거래 수: 40%
  - 기간 길이 (월 수): 20%
  - 시장 상태 커버리지 (상승/하락/횡보): 20%
  - 코인 다양성 (거래된 고유 코인 수): 20%

예시:
  - 1000건, 2년, 3개 regime, 100개 코인 → Highly Reliable
  - 1000건, 2개월, 1개 regime, 5개 코인 → Moderate (기간/다양성 부족)
```

### 2.2 p-value 기반 전략 유효성 검증

#### 2.2.1 귀무가설 설정

```
H0 (귀무가설): 이 전략의 수익률은 랜덤 진입과 동일하다.
H1 (대립가설): 이 전략의 수익률은 랜덤 진입보다 높다.

검정 방법: Permutation Test (비모수적)
  1. 실제 전략 수익률 R_actual 계산
  2. 시그널 타이밍을 N회(10,000회) 랜덤 셔플
  3. 각 셔플에서 수익률 R_shuffled 계산
  4. p-value = count(R_shuffled >= R_actual) / N
  5. p < 0.05 → 전략이 통계적으로 유의하게 우수
```

#### 2.2.2 사용자에게 표시하는 방법

```
전략 유효성: *** (p = 0.003)

*** p < 0.01  : 강한 통계적 유의성 (Very Strong)
**  p < 0.05  : 통계적 유의성 (Strong)
*   p < 0.10  : 약한 유의성 (Weak)
    p >= 0.10 : 유의하지 않음 (Not Significant)

주의: p-value는 "이 결과가 우연일 확률"이 아닙니다.
      "랜덤 전략이 같은 성과를 낼 확률"입니다.
```

#### 2.2.3 다중 검정 보정 (Multiple Testing Correction)

사용자가 10개 전략을 동시에 테스트하면 적어도 하나가 우연히 p < 0.05가 될 확률이 40%이다.

```
보정 방법: Bonferroni-Holm (Step-down)
  - 동시에 K개 전략 검증 시, 유의수준을 alpha / K로 조정
  - 5개 전략 동시 검증: 개별 p < 0.01이어야 전체 유의
  - 사용자에게 표시: "5개 전략 동시 검증 → 보정된 기준 적용됨"
```

### 2.3 Monte Carlo 시뮬레이션

#### 2.3.1 용도

- 전략 성과가 우연인지 구조적인지 판별
- 최악의 시나리오(Worst-case) 추정
- 신뢰 구간(Confidence Interval) 산출

#### 2.3.2 방법론

```
Monte Carlo 시뮬레이션 (거래 순서 셔플링):

1. 실제 거래 결과 리스트 [t1, t2, ..., tn] 추출
2. N회(10,000회) 반복:
   a. 거래 순서를 랜덤 셔플
   b. 셔플된 순서로 equity curve 재구성
   c. 최종 수익률, MDD, Sharpe 등 기록
3. 결과 분포에서:
   - 5th percentile: 최악의 시나리오
   - 50th percentile: 중앙값 (기대 수익)
   - 95th percentile: 최선의 시나리오
   - 최대 MDD 분포: 위험 구간

출력 예시:
  총 수익률: +18.5% (5th: +2.3%, 50th: +18.1%, 95th: +34.7%)
  최대 MDD: 12.3% (5th: 24.1%, 50th: 12.8%, 95th: 5.2%)
```

#### 2.3.3 사용자에게 표시

```
시뮬레이션 결과 (Monte Carlo 10,000회 기반):

  총 수익률: +18.5%
  신뢰 구간 (90%): [+2.3%, +34.7%]

  해석: 동일 전략을 동일 기간에 반복 적용 시,
        90%의 확률로 수익률이 +2.3% ~ +34.7% 범위에 있습니다.

  최대 손실 구간 (MDD):
  - 중앙값: 12.3%
  - 최악 (5th): 24.1%
  - 해석: 최악의 경우 자산의 24.1%까지 일시적 손실 가능
```

### 2.4 Walk-forward 검증 프레임워크

#### 2.4.1 목적

과거 전체 데이터로 최적화한 파라미터가 미래에도 유효한지 검증. autotrader에서 코인 필터링 과적합(2026-02-08)을 방지하는 핵심 메커니즘.

#### 2.4.2 PRUVIQ Walk-forward 절차

```
데이터: 2024-01-01 ~ 2026-02-15 (2년+)

Window 구성:
  - IS (In-Sample) 최적화 기간: 6개월
  - OOS (Out-of-Sample) 검증 기간: 3개월
  - Step: 3개월씩 이동

Walk-forward 윈도우:
  Round 1: IS 2024-01~2024-06 → OOS 2024-07~2024-09
  Round 2: IS 2024-04~2024-09 → OOS 2024-10~2024-12
  Round 3: IS 2024-07~2024-12 → OOS 2025-01~2025-03
  Round 4: IS 2024-10~2025-03 → OOS 2025-04~2025-06
  Round 5: IS 2025-01~2025-06 → OOS 2025-07~2025-09
  Round 6: IS 2025-04~2025-09 → OOS 2025-10~2025-12
  Round 7: IS 2025-07~2025-12 → OOS 2026-01~2026-02

각 Round에서:
  1. IS 기간에서 최적 파라미터 탐색 (Grid Search)
  2. 최적 파라미터를 OOS 기간에 적용
  3. OOS 성과 기록

Walk-forward 효율성 (WFE):
  WFE = OOS 평균 수익률 / IS 평균 수익률
  - WFE > 0.5: 양호 (파라미터가 미래에도 50%+ 유효)
  - WFE > 0.3: 보통 (주의 필요)
  - WFE < 0.3: 과적합 의심 (전략 재검토)
```

#### 2.4.3 사용자에게 표시

```
Walk-forward 검증 결과:

  IS 평균 수익률: +24.3%
  OOS 평균 수익률: +15.1%
  Walk-forward 효율성 (WFE): 62.1%

  판정: 양호 (파라미터가 미래에도 62% 효율로 작동)

  주의: WFE가 30% 미만이면 과적합 가능성이 높습니다.
        100% 이상이면 IS 기간이 비대표적일 수 있습니다.
```

## 3. 편향 방지 (Bias Prevention)

### 3.1 생존자 편향 (Survivorship Bias)

#### 3.1.1 문제 정의

현재 존재하는 코인만 시뮬레이션하면, 상폐되어 사라진 코인(LUNA, FTT 등)의 손실이 누락되어 성과가 과대 평가된다.

#### 3.1.2 PRUVIQ 대응: Coin Registry

ARCHITECTURE.md에 설계된 Coin Registry를 활용:

```
생존자 편향 방지 규칙:

1. 상폐 코인 데이터 보존
   - 상폐된 코인의 OHLCV 데이터를 삭제하지 않음
   - coin_registry.json에 delist 날짜 기록

2. 시뮬레이션 시 상폐 코인 포함
   - 2024년 백테스트 → 2024년에 존재했던 모든 코인 포함
   - 상폐된 코인은 상폐 시점까지만 시뮬레이션

3. 상폐 코인 강제 청산 처리
   - 상폐 시점에 보유 포지션 → 마지막 거래 가능 시점 close로 청산
   - 별도 exit_reason: "delisted"

4. 사용자에게 표시
   - "이 시뮬레이션에는 기간 중 상폐된 N개 코인이 포함되어 있습니다"
   - "생존자 편향 보정: 적용됨"
```

#### 3.1.3 생존자 편향 영향도 테스트

정기적으로(분기 1회) 다음 비교를 실행:
- A: 현재 활성 코인만으로 시뮬레이션
- B: 상폐 코인 포함 시뮬레이션
- 차이 = 생존자 편향 규모 추정
- 사용자에게 "생존자 편향 미보정 시 성과가 X% 과대 평가됩니다" 정보 제공

### 3.2 데이터 스누핑 편향 (Data Snooping Bias)

#### 3.2.1 문제 정의

동일 데이터셋으로 여러 파라미터를 시도하면, 우연히 잘 맞는 파라미터를 "최적"으로 오인한다. autotrader에서 2026-02-08 코인 필터링 과적합이 정확히 이 사례다.

#### 3.2.2 과적합 탐지 지표

**지표 1: IS/OOS 수익률 비교 (Overfitting Ratio)**

```
Overfitting Ratio (OR) = 1 - (OOS 수익률 / IS 수익률)

  OR < 0.2: 양호 (과적합 없음)
  OR 0.2-0.5: 주의 (경미한 과적합)
  OR 0.5-0.8: 경고 (심각한 과적합)
  OR > 0.8: 위험 (거의 완전한 과적합)
  OR > 1.0: OOS에서 손실 (과적합 확정)
```

**지표 2: Sharpe Ratio Decay (SRD)**

```
SRD = 1 - (OOS Sharpe / IS Sharpe)

IS Sharpe = 2.5, OOS Sharpe = 1.8 → SRD = 0.28 (양호)
IS Sharpe = 2.5, OOS Sharpe = 0.3 → SRD = 0.88 (위험)
```

**지표 3: Parameter Sensitivity Analysis**

```
최적 파라미터 주변 +-20% 범위에서 성과 변화:
  - 변화 < 10%: 안정적 (Robust)
  - 변화 10-30%: 보통 (Moderate)
  - 변화 > 30%: 불안정 (Fragile) → 과적합 의심

예시 (autotrader 교훈):
  SL 10%: PnL +$794
  SL 8%:  PnL -$200  ← 20% 변경에 수익→손실 전환 → 불안정
  SL 12%: PnL +$650  ← 양호
```

**지표 4: 그리드서치 결과 분포**

```
1000개 파라미터 조합 중 수익 조합 비율:
  - > 50%: 전략 자체가 강건 (대부분의 파라미터에서 수익)
  - 30-50%: 보통 (파라미터 의존성 존재)
  - 10-30%: 주의 (특정 파라미터에만 수익)
  - < 10%: 위험 (우연히 맞는 조합 찾은 것)
```

#### 3.2.3 과적합 경고 시스템

```
사용자가 그리드서치 결과를 볼 때 자동 표시:

  ┌──────────────────────────────────────────────────┐
  │  과적합 분석                                      │
  │                                                   │
  │  Overfitting Ratio: 0.45 (주의)                   │
  │  Sharpe Decay: 0.32 (양호)                        │
  │  Parameter Sensitivity: 25% (보통)                 │
  │  Grid Search 수익 비율: 38% (보통)                 │
  │                                                   │
  │  종합 판정: 경미한 과적합 가능성                   │
  │  권고: Walk-forward 검증 실행 권장                 │
  └──────────────────────────────────────────────────┘
```

### 3.3 시간대 편향 (Period Bias / Regime Bias)

#### 3.3.1 문제 정의

특정 시장 상황(상승장, 하락장, 횡보장)에서만 유효한 전략을 "항상 좋은 전략"으로 오인하는 문제. autotrader에서 2026-02-12에 발견: BB Squeeze SHORT는 바닥권 mean reversion에 구조적으로 취약하다.

#### 3.3.2 시장 상태 분류 (Regime Classification)

```
BTC 기준 시장 상태 분류:

  Bull (상승장): BTC 20일 수익률 > +10%
  Bear (하락장): BTC 20일 수익률 < -10%
  Sideways (횡보장): 그 외

각 전략에 대해 Regime별 성과를 분리 표시:

  BB Squeeze SHORT:
    Bull:     승률 55%, PF 1.2, 거래 120건
    Bear:     승률 72%, PF 3.1, 거래 85건   ← 하락장에 강함
    Sideways: 승률 60%, PF 1.8, 거래 95건

  Momentum LONG:
    Bull:     승률 65%, PF 2.5, 거래 90건   ← 상승장에 강함
    Bear:     승률 30%, PF 0.4, 거래 45건   ← 하락장에 약함
    Sideways: 승률 48%, PF 0.9, 거래 110건
```

#### 3.3.3 Regime 커버리지 경고

```
전략 백테스트 기간 중 Regime 분포:
  Bull: 8개월 (33%), Bear: 6개월 (25%), Sideways: 10개월 (42%)

경고 조건:
  - 특정 Regime < 15%: "이 전략은 [Regime]에서 충분히 검증되지 않았습니다"
  - 특정 Regime 0%: "경고: 이 전략은 [Regime]에서 전혀 검증되지 않았습니다"
```

#### 3.3.4 연도별 일관성 검증

```
일관성 점수 (Consistency Score):

  연도별 수익률: [+15%, +22%, +8%]
  표준편차: 7.1%
  평균: 15%
  변동계수 (CV): 47%

  CV < 30%: 매우 일관적 (Highly Consistent)
  CV 30-60%: 보통 (Moderate)
  CV 60-100%: 불일관적 (Inconsistent) → 경고
  CV > 100%: 매우 불일관적 (Highly Inconsistent) → 강한 경고
```

### 3.4 추가 편향 목록

| 편향 | 설명 | 탐지 방법 | PRUVIQ 대응 |
|------|------|----------|-------------|
| **Transaction Cost Bias** | 비용을 과소 추정하여 성과 부풀림 | 실거래 비용과 비교 | 보수적 비용 모델 (Taker 가정) |
| **Selection Bias** | 잘 맞는 코인만 선별 보고 | 전체 코인 집계 결과 필수 표시 | "전체 N개 코인 중 수익 M개" |
| **Confirmation Bias** | 원하는 결과만 보려는 경향 | 자동화된 검증 (사람 판단 배제) | QA 파이프라인 자동 실행 |
| **Execution Bias** | 시뮬레이션 체결가와 실제 체결가 차이 | 슬리피지 모델링 | 슬리피지 0.02% 기본 적용 |
| **Roll-over Bias** (선물) | 선물 만기 교체 시 가격 불연속 | 무기한 선물 사용 | USDT 무기한 선물 전용 |

## 4. 사용자 투명성 (User Transparency)

### 4.1 법적 면책 조항

#### 4.1.1 필수 면책 조항 (모든 시뮬레이션 결과 페이지에 표시)

```
DISCLAIMER / 면책 조항

1. 이 시뮬레이션 결과는 과거 데이터에 기반한 가상의 성과입니다.
2. 과거 성과는 미래 수익을 보장하지 않습니다.
3. 실제 거래에서는 슬리피지, 유동성 부족, 거래소 장애 등으로
   시뮬레이션과 다른 결과가 발생할 수 있습니다.
4. 이 플랫폼은 투자 조언을 제공하지 않습니다.
5. 암호화폐 거래는 원금 전액 손실 위험이 있습니다.
6. 거래 결정에 대한 책임은 전적으로 사용자에게 있습니다.

PRUVIQ는 이 시뮬레이션 결과를 기반으로 한 어떠한 투자 손실에 대해서도
법적 책임을 지지 않습니다.
```

#### 4.1.2 시뮬레이션 한계 명시 (상세)

```
시뮬레이션의 알려진 한계:

1. 유동성 가정
   - 시뮬레이션은 항상 원하는 가격에 체결된다고 가정합니다.
   - 실제로는 유동성 부족으로 슬리피지가 발생할 수 있습니다.
   - 특히 소형 코인에서 큰 포지션 시 체결이 어려울 수 있습니다.

2. 시장 영향
   - 시뮬레이션은 자신의 거래가 시장에 영향을 미치지 않는다고 가정합니다.
   - 실제로 대규모 주문은 가격을 이동시킵니다.

3. 데이터 한계
   - 1시간봉 기준: 봉 내부 가격 경로를 알 수 없습니다.
   - SL/TP 동시 도달 시 보수적 처리 (SL 우선)를 적용합니다.
   - 거래소 장애, 네트워크 지연 등은 반영하지 않습니다.

4. 비용 모델 한계
   - 고정 슬리피지 0.02% 적용 (실제는 변동적)
   - 펀딩비는 히스토리 기반 적용 (미래 펀딩비 예측 불가)
   - VIP 등급 수수료 할인 미반영
```

## 5. 자동화된 QA 파이프라인 (Automated QA Pipeline)

### 5.1 새 전략 추가 시 자동 검증 체크리스트

전략이 Strategy Protocol을 구현하고 시스템에 등록될 때, 다음 체크리스트가 자동 실행된다.

```
┌──────────────────────────────────────────────────────────────────┐
│  Strategy QA Pipeline (새 전략 추가 시 자동 실행)                  │
│                                                                   │
│  Phase 1: 구조 검증 (Structural Validation)                       │
│  ─────────────────────────────────────────                        │
│  [ ] Protocol 준수: name, calculate_indicators, check_signal,     │
│      get_params 메서드 존재 확인                                   │
│  [ ] 타입 힌트 일치: 반환 타입이 Protocol과 일치                   │
│  [ ] 금지 패턴 없음: shift(-n), center=True, bfill 미사용        │
│  [ ] get_params()가 모든 __init__ 파라미터를 포함                  │
│                                                                   │
│  Phase 2: Look-ahead Bias 검증                                    │
│  ──────────────────────────────                                    │
│  [ ] Test A: 시간 순서 검증 (entry < exit)                        │
│  [ ] Test B: 진입가 일관성 (entry = next open)                    │
│  [ ] Test C: 미래 데이터 접근 차단 (truncated df 테스트)           │
│  [ ] Test D: 랜덤 전략 수익률 < 0 확인                            │
│  [ ] Test E: 시프트 패턴 분석 (코드 정적 분석)                    │
│                                                                   │
│  Phase 3: 기본 성과 검증 (Sanity Check)                           │
│  ──────────────────────────────────────                            │
│  [ ] 1개 코인(BTCUSDT)에서 실행 가능 확인                         │
│  [ ] 10개 코인에서 배치 실행 가능 확인                             │
│  [ ] 거래 0건이 아닌지 확인 (최소 1건 이상)                       │
│  [ ] 결과 필드 누락 없음 (SimulationResult 완전성)                │
│  [ ] 비용 차감 후 결과 반환 확인                                   │
│                                                                   │
│  Phase 4: 재현성 검증                                              │
│  ──────────────────                                                │
```

### 5.2 데이터 업데이트 후 결과 Drift 감지

#### 5.2.1 Drift 정의

데이터가 업데이트(새 캔들 추가)된 후 기존 결과가 변경되는 경우.

```
Drift 유형:

  Type 1: 정상 Drift
    - 새 데이터 추가로 인해 거래 수 증가 → 결과 소폭 변동
    - 허용 범위: 기존 결과 대비 +-5%

  Type 2: 비정상 Drift
    - 과거 데이터 수정으로 인해 기존 거래 결과 변경
    - 허용 범위: 0% (과거 거래는 변경 불가)
    - 원인: 데이터 소스 오류, 거래소 OHLCV 정정

  Type 3: 코드 Drift
    - 엔진/전략 코드 변경으로 인해 동일 데이터에서 다른 결과
    - 허용 범위: 0% (의도적 변경 제외)
    - 원인: 버그 수정, 로직 변경
```

#### 5.2.2 Drift 감지 파이프라인

```
매일 시뮬레이션 배치 실행 시 (UTC 01:00):

1. 캐시된 이전 결과 로드
2. 새 데이터로 시뮬레이션 재실행
3. 비교:
   - 과거 기간(동일 시점까지)의 거래가 동일한지 확인
   - 새 기간의 거래만 추가되었는지 확인
4. Type 2 Drift 감지 시:
   - 즉시 경고 (Telegram 알림)
   - 원인 분석: 데이터 변경 diff 확인
   - 영향 평가: 변경된 거래 수와 PnL 차이
5. 결과 업데이트 및 캐시 갱신
```

### 5.3 리그레션 테스트 (Regression Testing)

#### 5.3.1 테스트 레벨

```
Level 1: Unit Tests (전략/엔진 단위)
  - 각 전략의 check_signal 정확성
  - 엔진의 SL/TP/TIMEOUT 처리 정확성
  - 비용 모델 계산 정확성
  - 실행 시간: < 30초

Level 2: Integration Tests (엔진 + 전략 + 데이터)
  - 10개 코인으로 E2E 시뮬레이션
  - 배치 실행기 + 캐시 + 결과 집계
  - 실행 시간: < 5분

Level 3: Regression Tests (전체 코인 + 결과 비교)
  - Golden Dataset: 고정된 테스트 데이터셋 (BTCUSDT 2024-01~2024-12, 8760 행)
  - Golden Result: 해당 데이터에서의 기대 결과 (거래 수, 승률, PnL 정확 일치)
  - 코드 변경 시 Golden Result와 비교 → 불일치 시 실패
  - 실행 시간: < 30초

Level 4: Smoke Tests (프로덕션 환경)
  - 실제 데이터로 1개 전략 실행
  - 결과가 합리적 범위 내인지 확인
  - API 엔드포인트 응답 확인
  - 실행 시간: < 2분
```
